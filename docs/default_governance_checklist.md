# Default Governance Checklist

每次 ChatGPT / Codex 任务开始前，先快速过这 5 问。能一句话回答就不要展开成长报告。默认不必逐条展示 5 问；除非任务高风险或用户要求，只在必要时用 3-5 句话说明判断结果。

## 1. 是否值得现在做？

- 这件事是否服务当前目标？
- 影响是否大于成本？
- 如果只是好奇、低价值或高努力低影响，先放入 backlog。

## 2. 任务类型是什么？

标记一个主类型：

- `research`：只查、读、比较，不写入正式产物。
- `proposal`：给方案、结构、清单，不执行。
- `build`：创建或修改文件、代码、文档。
- `install`：安装 Skill、插件、依赖或工具。
- `publish`：提交、推送、部署、公开发布。

`install` / `publish` / `delete` / `external write` 默认必须用户确认，除非用户在当前任务中明确授权。

## 3. 关键假设和证据缺口是什么？

- 哪些是用户明确给出的事实？
- 哪些只是推断？
- 哪些缺证据会影响结果？
- 涉及用户心理、付费转化、安全、版权、隐私时，先标出假设，不把猜测写成结论。

## 4. 最小可逆路径是什么？

- 有没有更小、更简单、可回滚的第一步？
- 能用模板就不先做插件。
- 能先 research 就不直接 build。
- 能人工确认就不自动循环。
- 只读取完成任务必需的最小上下文。

## 5. 完成标准、回滚方式、是否需要用户确认？

- 完成后应出现什么文件、结果或验证证据？
- 如果做错，怎么撤回？
- 是否涉及安装、删除、发布、凭据、隐私、正式目录、外部写入或长期架构？
- 如果是，先暂停并回到用户确认。

## 默认规则

安全、隐私、版权、证据不足优先于效率。用户明确说“只调研”时，不进入 `build`、`install` 或 `publish`。

## 停止条件

如果任务目标不清、证据不足会导致误写入、需要安装/发布但没有授权，或发现当前动作偏离用户最新请求，立即停止，转为提问、风险说明或只输出 proposal。
